# Test Isolation in Cypress

**Test isolation** is a Cypress feature that runs every test in a clean browser
context. It clears the page, cookies, local storage, and session storage before
each test, so each test starts from a known state and can pass independently of
every other test. It is enabled by default and is one of the most effective ways
to prevent flaky, order-dependent tests.

## Why test isolation matters

A test is only valuable if you can trust its result. The fastest way to lose
that trust is to let one test leak state into the next: a cookie left behind, a
logged-in session that carries over, a piece of `localStorage` that quietly
changes how the next test behaves. When that happens, a passing suite can hide
real bugs, and a failing test can point at the wrong cause.

Test isolation is how Cypress protects that trust. By starting every test from a
clean, known state, Cypress gives you tests that are:

- **Reliable**: each test sets up exactly the state it needs, so it passes for
  the same reason every time instead of relying on a side effect of the test
  before it. This is the single most effective defense against flaky tests.
- **Deterministic**: because no state carries over, tests can run in any order,
  be randomized, retried, or run in parallel and still produce the same result.
  The order your tests happen to be written in stops being a hidden dependency.
- **Easy to debug**: when a test fails, the cause is contained within that
  test. You are not hunting through earlier tests to figure out what state they
  left behind, which dramatically shortens the path from failure to fix.
- **Trustworthy as documentation**: each test reads as a complete, standalone
  description of a behavior, because it includes everything required to reach
  the state it verifies.

## How test isolation works

Cypress delivers this clean starting point in two layers, applied _in between_
each test so the operation of one test cannot affect another.

First, Cypress starts each test with a clean test slate by restoring and
clearing all:

- [aliases](/api/commands/as)
- [clock mocks](/api/commands/clock)
- [intercepts](/api/commands/intercept)
- [spies](/api/commands/spy)
- [stubs](/api/commands/stub)
- [viewport changes](/api/commands/viewport)
- extra browser tabs and windows opened during the test

Second, Cypress runs each test in a clean browser context so that the
application under test behaves consistently every time it runs. To do this,
Cypress resets the browser context by:

- navigating to `about:blank` to clear the current page and DOM (end-to-end
  testing)
- unmounting the rendered component to clear the DOM (component testing)
- clearing [cookies](/api/cypress-api/cookies) in all domains
- clearing
  [`localStorage`](https://developer.mozilla.org/en-US/docs/Web/API/Window/localStorage)
  in all origins
- clearing
  [`sessionStorage`](https://developer.mozilla.org/en-US/docs/Web/API/Window/sessionStorage)
  in all origins

This browser-context behavior is what the `testIsolation` configuration
controls.

## Configuring test isolation

Test isolation is controlled by the
[`testIsolation`](/app/references/configuration#e2e) configuration option, which
accepts a boolean value of `true` or `false` and defaults to `true`.

In end-to-end testing, you can set it globally in your Cypress configuration and
override it for a suite at the `describe` level, and only there. Overriding
`testIsolation` on an individual test (`it`) is not supported.

In component testing, test isolation is always on and cannot be disabled. See
[Test isolation in component testing](#Test-isolation-in-component-testing)
below.

## Writing isolated tests

Test isolation is enabled by default and is the recommended setting. Because each
test starts in a fresh browser context, write each test so it establishes
everything it needs and never depends on a previous test. A few practices keep
your tests isolated and reliable:

- **Set up state in hooks, not in earlier tests.** Use a `beforeEach` hook to
  establish the state each test relies on, so every test is self-contained and
  passes whether run on its own or in a randomized order.
- **Re-visit the application in each test.** Since the page is reset between
  tests, call [`cy.visit()`](/api/commands/visit) (typically in a `beforeEach`)
  before interacting with your app.
- **Cache expensive setup with `cy.session()`.** Repeating a slow flow such as
  logging in for every test is wasteful.
  [`cy.session()`](/api/commands/session) caches and restores cookies and
  storage so you can skip that repetition while keeping each test isolated. Call
  [`cy.visit()`](/api/commands/visit) after it, since the page still starts
  blank.
- **Do not rely on the order tests run in.** If a test passes only because of
  something an earlier test did, it is not truly isolated. See
  [Having Tests Rely On The State Of Previous Tests](/app/core-concepts/best-practices#Having-Tests-Rely-On-The-State-Of-Previous-Tests).

## Keeping state without disabling isolation

Because Cypress clears all cookies, `localStorage`, and `sessionStorage` before
each test when test isolation is enabled, any state your application writes to
the browser is gone by the next test. The following approaches let you keep the
state you need while preserving the isolation that keeps your tests independent.
If you instead need tests to share state with one another, see
[Disabling test isolation](#Disabling-test-isolation).

### Re-set a known cookie in `beforeEach`

If you know the cookie value ahead of time (for example, a cookie-consent
acceptance or a feature flag), use [`cy.setCookie()`](/api/commands/setcookie)
inside a `beforeEach` hook so the cookie is present at the start of every test.
Each test still begins from a clean, explicit state, so isolation is fully
preserved:

```javascript
// cypress/support/e2e.js  (or inside a describe block)
beforeEach(() => {
  cy.setCookie('cookieConsent', 'accepted')
})
```

### Cache server-set cookies with `cy.session()`

If your cookie is generated by the server after a login, use
[`cy.session()`](/api/commands/session) to capture the full browser context once
and restore it before each subsequent test, without repeating the login flow.
This keeps tests both fast and isolated, since every test still starts from the
same known session:

```javascript
// cypress/support/commands.js
Cypress.Commands.add('login', (username, password) => {
  cy.session([username, password], () => {
    cy.visit('/login')
    cy.get('[data-test=username]').type(username)
    cy.get('[data-test=password]').type(password)
    cy.get('form').contains('Log In').click()
    cy.url().should('contain', '/dashboard')
  })
})

// In your spec or support file
beforeEach(() => {
  cy.login('myuser', 'password123')
  cy.visit('/dashboard')
})
```

`cy.session()` restores all cookies (and localStorage / sessionStorage) that
were present when the session was first created, so server-set auth cookies are
available in every test without a full login round-trip.

## Disabling test isolation

If a suite genuinely needs to share state between tests, you can disable test
isolation for end-to-end testing by setting `testIsolation: false`, either
globally or at the `describe` level.

### What changes when you disable it

Setting `testIsolation: false` turns off exactly one thing: the browser-context
reset between tests. When it is disabled, Cypress no longer:

- clears [cookies](/api/cypress-api/cookies) in all domains
- clears
  [`localStorage`](https://developer.mozilla.org/en-US/docs/Web/API/Window/localStorage)
  in all origins
- clears
  [`sessionStorage`](https://developer.mozilla.org/en-US/docs/Web/API/Window/sessionStorage)
  in all origins
- navigates to `about:blank` between tests, so the application under test,
  including its current page and DOM, stays loaded going into the next test

As a result, this browser-level state carries over from one test to the next
within that suite.

There is one nuance to the page reset: it is decided by the _next_ test. If a
test with isolation off is followed by a test that has isolation on, Cypress
will still navigate to `about:blank` first so the isolated test starts clean.
The page only persists when the following test also has isolation off.

Additionally, the [`cy.session()`](/api/commands/session) command will still
clear the browser context (cookies and storage) when it establishes a session,
but it will no longer clear the current page as part of that process.

### Example

Turn off test isolation for a specific `describe` block by setting
`testIsolation: false`. State set in a `before` hook then persists across all
tests in that suite:

```javascript
describe('Marketing pages', { testIsolation: false }, () => {
  before(() => {
    cy.setCookie('cookieConsent', 'accepted')
    cy.visit('/home')
  })

  it('shows the home page', () => {
    // cookieConsent cookie is present; page is already loaded
  })

  it('shows the about page', () => {
    cy.visit('/about')
    // cookieConsent cookie is still present
  })
})
```

### Trade-offs and using it safely

The upside of disabling test isolation is that tests in a suite can build on each
other. You can log in once in the first test and have later tests start already
authenticated and on the right page, without re-visiting or re-establishing state
each time. For long, linear flows this can be faster and less repetitive.

The risk is that tests become order-dependent. Because state leaks forward, a
test can quietly start relying on something a previous test did. That creates two
real hazards:

1. A test may pass only because of the test before it. Run it on its own and it
   fails.
2. A failure early in the suite can cascade, causing later tests to fail for
   reasons that have nothing to do with what they are actually testing, making
   the real problem harder to find.

If you choose to disable it, use it deliberately and safely:

- Treat a `testIsolation: false` `describe` block as one connected sequence, and
  order the tests intentionally.
- Verify each test still passes on its own. Temporarily isolate it with
  `.only()` and run it; if it fails alone, it has a hidden dependency on a
  previous test, the exact problem test isolation is designed to prevent.
- Narrow the scope: apply `testIsolation: false` to the specific `describe`
  block that needs it rather than globally.
- If you only need cookies or storage to persist (not the page), prefer the
  approaches in
  [Keeping state without disabling isolation](#Keeping-state-without-disabling-isolation),
  which give you the same speedup while keeping tests independent.

## Test isolation in component testing

Test isolation is always enabled in component testing and cannot be turned off or
configured. As described in [How test isolation works](#How-test-isolation-works),
Cypress isolates each component test by unmounting the rendered component and
clearing cookies, `localStorage`, and `sessionStorage` before every test. This
guarantees every component test renders into a clean context and verifies your
component's behavior without interference from a previous test.

## Frequently Asked Questions

### What is test isolation in Cypress?

Test isolation is the practice of running each test in a clean browser context.
Before every test, Cypress clears the page, cookies, local storage, and session
storage so the test starts from a known state and does not inherit anything from
the test that ran before it.

### Does test isolation prevent flaky tests?

It prevents one of the most common causes of flaky tests: state leaking from one
test into the next. By guaranteeing a clean starting point, test isolation makes
tests deterministic so they pass for the same reason every time, in any order.

### Is test isolation enabled by default?

Yes. The [`testIsolation`](/app/references/configuration#e2e) option defaults to
`true` for end-to-end testing, and it is always on for component testing.

### Can I disable test isolation?

Yes, for end-to-end testing. Set `testIsolation: false` either globally in your
Cypress configuration or at the suite (`describe`) level. Disabling it lets state
persist across tests, so do this only when you understand the trade-offs.

### Can I set test isolation on an individual test?

No. The `testIsolation` option can only be overridden at the suite (`describe`)
level. Setting it on an individual test (`it`) is not allowed and cannot be done.

### What values does the `testIsolation` option accept?

Only the boolean values `true` or `false`. It defaults to `true`, and any other
value is invalid and will be rejected by Cypress.

### Is test isolation available in component testing?

Test isolation is always enabled in component testing and cannot be configured.
Every component test renders into a clean browser context.
