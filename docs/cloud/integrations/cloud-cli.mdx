# <Icon name="terminal" /> Cloud CLI: Debug from your terminal

The Cypress Cloud CLI (`cy-cloud`) brings your Cypress Cloud data to the command line. Check the status of a CI run, pull the screenshot of a failed test, or replay the exact commands, network calls, and console logs that led to a failure, all without opening a browser.

Every command returns structured JSON, so the CLI is equally at home in a shell script, a CI job, or an [AI coding agent](#Use-with-AI-agents).

:::info

<WhatYoullLearn />

- How the Cloud CLI helps you debug CI failures without leaving your terminal
- How to install and authenticate `cy-cloud`
- The full command reference for orgs, projects, runs, specs, tests, and replays
- A step-by-step workflow for triaging a failing run
- How caching, rate limits, output schemas, and configuration work

:::

:::note

The Cloud CLI is available on **every Cypress Cloud plan**, including the free Starter plan, at no additional cost. An organization admin enables it once from the **Integrations** page.

:::

## <Icon name="circle-question" /> Why the Cloud CLI

When a test fails in CI, the friction is rarely the fix. It is the triage: finding the run, opening the right test, reading the error, and reconstructing what the app was doing at the moment it failed. That usually means leaving your editor, signing in to Cypress Cloud, and clicking through a run.

The Cloud CLI removes that round trip. It exposes the same Cloud data you would click through, as commands you can pipe, script, and compose:

- **Stay in your terminal.** List failing runs, drill into failing tests, and read errors and stack traces from the shell.
- **Replay without a browser.** Query the [Test Replay](/cloud/features/test-replay) timeline for a specific attempt, scoped to the commands, network requests, and logs around the point of failure.
- **Automate it.** Because output is JSON, you can wire the CLI into scripts, dashboards, and custom tooling.
- **Hand it to an agent.** The CLI is designed for both humans and AI agents, so your coding assistant can gather the same context you would.
- **Grab a failure screenshot.** Save the screenshot Cypress captured at the moment a test failed straight to disk.

## <Icon name="gear" /> How it works

The CLI authenticates you against Cypress Cloud, then calls the Cypress Cloud API on your behalf. Data you retrieve is scoped to the projects and organizations your Cypress Cloud user can already see.

1. **Authenticate** once with `cy-cloud login` (OAuth) or a personal access token.
2. **Query** your Cloud data with commands like `run list`, `test get`, and `replay timeline`.
3. **Receive JSON** on standard output, ready to read, pipe, or parse.

Replay data is downloaded once and [cached locally](#Caching), so repeated queries against the same test are fast and do not re-download data.

## <Icon name="list-check" /> Requirements

To use the Cloud CLI you must:

- Have a [Cypress Cloud](https://cloud.cypress.io) account.
- Have the Cloud CLI integration enabled for your organization. An organization admin enables it from the **Integrations** page in Cypress Cloud.
- Be running [Node.js](https://nodejs.org) `22.21.0` or later.

:::note

The Cloud CLI reads recorded run data. Your tests still run and record the same way they do today, using `cypress run --record`. See [Recording Cloud Runs](/cloud/features/recorded-runs) if you have not set up recording yet.

:::

## <Icon name="download" /> Install

Install `@cypress/cloud` globally to get the `cy-cloud` command:

<PackageManagerTabs install="@cypress/cloud" global />

Confirm the install and see every available command:

```bash
cy-cloud --help
```

Every command and subcommand also supports `--help`, so you can explore the CLI as you go:

```bash
cy-cloud run list --help
```

## <Icon name="lock" /> Authenticate

You can authenticate in three ways. OAuth is recommended for local use; a token in an environment variable is best for CI.

### OAuth (recommended)

Running `login` with no arguments starts a browser-based sign-in. It opens Cypress Cloud in your browser, you approve the request, and your session is stored securely on your machine. OAuth sessions refresh automatically, so you do not manage token expiry yourself.

```bash
cy-cloud login
```

If the browser does not open automatically, the CLI prints a URL you can open manually. You can set how many seconds the CLI waits for the sign-in to complete (the default is 120 seconds):

```bash
cy-cloud login --timeout 180
```

### Personal access token

If you prefer a token, or your environment cannot open a browser, pass a personal access token to `login`. The CLI checks the token with Cypress Cloud and, if it is valid, saves it to a credentials file (`~/.config/cy-cloud/auth.json`, readable only by your user) so later commands reuse it without you passing it each time:

```bash
cy-cloud login --token <YOUR_PERSONAL_ACCESS_TOKEN>
```

### Environment variable (best for CI)

Set `CYPRESS_CLOUD_TOKEN` to authenticate without a `login` step and without writing any credentials to disk. This takes precedence over any stored credentials, which makes it ideal for CI and other non-interactive environments.

<OsTabs>
  <TabItem value="macos" label="macOS / Linux">

```bash
export CYPRESS_CLOUD_TOKEN=<YOUR_PERSONAL_ACCESS_TOKEN>
cy-cloud run list --projectId <projectId>
```

  </TabItem>
  <TabItem value="windows" label="Windows">

```shell title="Command Prompt"
set CYPRESS_CLOUD_TOKEN=<YOUR_PERSONAL_ACCESS_TOKEN>
cy-cloud run list --projectId <projectId>
```

```shell title="PowerShell"
$env:CYPRESS_CLOUD_TOKEN = "<YOUR_PERSONAL_ACCESS_TOKEN>"
cy-cloud run list --projectId <projectId>
```

  </TabItem>
</OsTabs>

:::note

`cy-cloud logout` clears credentials stored by `login`. It does not unset `CYPRESS_CLOUD_TOKEN`. If that variable is set, you remain authenticated through it until you unset it.

:::

Check your current authentication state at any time:

```bash
cy-cloud status
```

`"authenticated": true` is the signal that you are signed in. `method` is `oauth` or `pat`, and `expiresAt`, when present, is when an OAuth session refreshes:

```json
{
  "authenticated": true,
  "method": "oauth",
  "expiresAt": "2026-07-21T18:30:00.000Z"
}
```

If you are not signed in, `authenticated` is `false` and the other fields are omitted.

## Command reference

Most data commands live under a top-level noun (`org`, `project`, `run`, `spec`, `test`, `replay`, `cache`), each with subcommands such as `list` and `get`. For example, `cy-cloud run list` lists runs and `cy-cloud run get` fetches one run. Authentication and utility commands (`login`, `logout`, `status`, `version`) sit at the top level. Each data command returns JSON on standard output.

| Command             | What it does                                              |
| ------------------- | --------------------------------------------------------- |
| `cy-cloud login`    | Sign in to Cypress Cloud.                                 |
| `cy-cloud logout`   | Clear stored credentials.                                 |
| `cy-cloud status`   | Check your authentication state and Cloud status.         |
| `cy-cloud version`  | Print the CLI version and whether an upgrade is required. |
| `cy-cloud org list` | List the organizations you belong to.                     |
| `cy-cloud project`  | List projects, or get details for one project.            |
| `cy-cloud run`      | List runs for a project, or get details for one run.      |
| `cy-cloud spec`     | List specs, or get details for one spec.                  |
| `cy-cloud test`     | List tests, or get details for one test.                  |
| `cy-cloud replay`   | Get Test Replay info and query the replay timeline.       |
| `cy-cloud cache`    | Inspect and clear the local Test Replay cache.            |

Most `list` commands accept `--page` and `--limit` for pagination. For example, `--page 2 --limit 25` returns the second page of 25 results, and every list response includes a `pagination` object with the current `page`, `perPage`, and the `total` number of matching results. Filters that take multiple values (`--status`, `--branch`, `--specId`, `--testId`, `--orgId`) accept a comma-separated list, such as `--status passed,failed`.

```bash
cy-cloud logout
```

```json
}
```

### `cy-cloud project list`

List every project you can access:

```bash
cy-cloud project list
```

Each project includes its `projectId` and display name:

```json
{
  "projects": [
    { "projectId": "abc123", "name": "design-system" },
    { "projectId": "def456", "name": "storefront" }
  ],
  "pagination": { "page": 1, "perPage": 30, "total": 2 }

```

Or list projects in specific organizations, with pagination:

```bash

```

### `cy-cloud project get`

Get details for a single project by its project ID (the `projectId` from your Cypress [configuration](/cloud/account-management/projects)):

```bash
cy-cloud project get <projectId>
```

The response also includes the project's latest run numbers:

```json
{
  "projectId": "abc123",
  "name": "design-system",
  "latestRunId": 4021,
  "latestCompletedRunId": 4020
}
```

### `cy-cloud run list`

List runs for a project. Filter by branch and status to find the run you care about, for example the most recent failed run on `main`:

```bash
cy-cloud run list --projectId <projectId> --branch main --status failed --limit 1
```

Each run includes its number, branch, timestamps, and status:

```json
{
  "runs": [
    {
      "runNumber": 4021,
      "branch": "main",
      "createdAt": "2026-07-21T02:14:05.000Z",
      "completedAt": "2026-07-21T02:19:44.000Z",
      "status": "failed"
    }
  ],
  "pagination": { "page": 1, "perPage": 30, "total": 87 }
}
```

`--status` accepts `passed`, `failed`, `running`, `errored`, `timedOut`, `cancelled`, and `noTests` (comma-separate to combine).

### `cy-cloud run get`

Get details for a specific run by project ID and run number:

```bash
cy-cloud run get <projectId> <runNumber>
```

The details add the commit SHA, per-status test counts, and duration in milliseconds:

```json

```

Save the failure screenshot for a test. Pass `--screenshot` on its own to write to a temporary directory, or give it a path to choose the destination (for example, a `screenshots` directory):

```bash
cy-cloud test get <testId> --screenshot ./screenshots
```

```json
{
```

### `cy-cloud replay info`

[Test Replay](/cloud/features/test-replay) captures the commands, network traffic, and console logs for a test. The `replay` commands let you query that timeline for a single test attempt.

:::note

Replay attempts are **1-indexed to match the Cypress Cloud dashboard**, where the first run of a test is Attempt 1. When you omit `--attempt`, replay commands default to the last (most recent) attempt, which is the final retry for a test that was retried.

:::

:::caution

If a test's replay data is not available yet (for example, it is still processing after the run completes) or was never captured, `replay` commands return an error like `Failed to download replay: Replay "<testId>" not available`. The CLI does not wait or poll, so re-run the command once processing has finished. If it keeps failing, confirm [Test Replay](/cloud/features/test-replay) was enabled for the run and that it is within your data retention window.

:::

Get a summary of a replay, including how many command, network, and log events it holds and where the failure occurred:

```bash
cy-cloud replay info --testId <testId>
```

The summary reports the event counts and where the failure occurred:

```json
{
  "runnableId": "7c1e9d20-5a3b-4c8d-9e0f-1a2b3c4d5e6f",
  "attemptId": 1,
  "commandEvents": 214,
  "networkEvents": 38,
  "logs": 12,
  "failedAt": 1753063204120
}
```

### `cy-cloud replay timeline`

Query the replay timeline. By default the timeline is broad, so scope it to what you need. Include `--commands`, `--logs`, and `--network` to choose event types, and narrow to the failure with `--failedOnly` or `--aroundFailure`.

Get the five commands leading up to the failure, plus the failed command:

```bash
cy-cloud replay timeline --testId <testId> --commands --aroundFailure 5
```

Events come back in a single `events` array. Each carries a `category` of `command`, `network`, or `log`, so you can tell them apart:

```json
{
  "events": [
    {
      "category": "command",
      "commandId": "c-118",
      "type": "assertion",
      "name": "assert",
      "runnableId": "7c1e9d20-5a3b-4c8d-9e0f-1a2b3c4d5e6f",
      "attempt": 1,
      "startTimestamp": 1753063204120,
      "endTimestamp": 1753063208120,
      "state": "failed",
      "message": "expected '.cart-total' to have text '$45.00'",
      "hookId": null
    }
  ]
}
```

Or include command, network (XHR and Fetch), and console log events for an attempt:

```bash
cy-cloud replay timeline --testId <testId> --commands --network --logs
```

The `events` array now also holds `network` and `log` entries alongside the commands, each tagged by its `category`:

```json
{
  "events": [
    {
      "category": "network",
      "id": "n-42",
      "timestamp": 1753063205000,
      "url": "https://api.example.com/cart",
      "method": "POST",
      "request": { "headers": { "content-type": "application/json" } },
      "response": {
        "status": 500,
        "headers": { "content-type": "application/json" }
      }
    },
    {
      "category": "log",
      "id": 7,
      "type": "console",
      "timestamp": 1753063205200,
      "payload": "Uncaught TypeError: Cannot read properties of undefined (reading 'total')"
    }
  ]
}
```

Useful `replay timeline` options:

| Option               | Description                                                               |
| -------------------- | ------------------------------------------------------------------------- |
| `--testId`           | The test to replay (required).                                            |
| `--attempt`          | Attempt number (1-indexed). Defaults to the last (most recent) attempt.   |
| `--commands`         | Include Cypress command events.                                           |
| `--network`          | Include network events by type, comma-separated. Defaults to `XHR,Fetch`. |
| `--logs`             | Include console log events.                                               |
| `--failedOnly`       | Return only failed command events.                                        |
| `--aroundFailure`    | Return N commands before the failure, plus the failed command.            |
| `--page` / `--limit` | Paginate through the timeline.                                            |

`--network` accepts standard resource types, including `XHR`, `Fetch`, `Document`, `Script`, `Stylesheet`, `Image`, `Font`, `WebSocket`, and `All`.

### `cy-cloud cache info`

Print the local Test Replay cache location, file count, and total size on disk.

```bash

```

```json
{
  "location": "/Users/you/.cache/cy-cloud",
  "files": 12,
  "bytes": 48210432
}
```

### `cy-cloud cache clear`

Remove everything from the cache. Prints `{ "success": true }` when done.

```bash

```

### `cy-cloud cache cleanup`

Delete expired and oversized entries. This runs automatically as a maintenance task after commands, but you can trigger it manually. See [Caching](#Caching) for the size and age limits.

```bash

```

## Workflow: triage a failing run

This walkthrough puts the commands together on a common situation: the nightly run on `main` failed, and a checkout test is red. It takes you from a failing run to the exact cause, without opening a browser.

**1. Find the failing run.** Grab the most recent failed run on `main` and note its run number.

```bash
cy-cloud run list --projectId <projectId> --branch main --status failed --limit 1
```

**2. List the failing tests in that run.**

```bash
cy-cloud test list --projectId <projectId> --runNumber <runNumber> --status failed
```

**3. Read the failure.** Get the failing test's error and stack trace, and save its screenshot so you can see the app at the moment it broke.

```bash
cy-cloud test get <testId> --screenshot ./screenshots
```

**4. See what the replay holds.** Check the replay summary to confirm where the failure occurred and how much detail is captured.

```bash
cy-cloud replay info --testId <testId>
```

**5. Reconstruct the moment of failure.** Pull the commands right before the failure, plus the network calls and console logs for the attempt. This is the same context you would scrub through in Test Replay, delivered as JSON.

```bash
cy-cloud replay timeline --testId <testId> --commands --aroundFailure 5 --network --logs
```

At this point you have the error, the screenshot, the commands that led up to it, and the network and console activity around it, all in your terminal. From here you can fix the test, or hand the JSON to an [agent](#Use-with-AI-agents) and ask it to propose a fix.

:::tip

Wrap this sequence in a shell script that takes a project ID and branch, and you have a one-command "why did the last build fail" tool for your team.

:::

## <Icon name="code" /> Output format and schemas

Every data command prints JSON to standard output, so you can pipe results into other tools. For example, print the run numbers of the last five failed runs on `main` with [`jq`](https://jqlang.github.io/jq/):

```bash
cy-cloud run list --projectId <projectId> --branch main --status failed --limit 5 \
  | jq '.runs[].runNumber'
```

To see the exact shape of a command's output before you parse it, add `--schema`. It prints the Zod-compliant JSON schema for that command's output and makes no network call:

```bash
cy-cloud test get --schema
```

It returns a JSON Schema describing the output fields and their types, which begins like this (abbreviated):

```json
{
  "type": "object",
  "properties": {
    "testId": { "type": "string", "format": "uuid" },
    "status": { "type": "string" },
    "attempt": { "type": "integer" },
    "state": { "type": "string" },
    "failureScreenshotPath": { "type": "string" }
  },
  "required": ["testId", "status", "attempt", "state"]
}
```

This is especially useful when generating parsing code or briefing an agent on what to expect.

## <Icon name="database" /> Caching

Replay data can cover an entire spec, so the CLI downloads a replay database once and caches it locally. Later `replay` queries for any test in that database read from the cache instead of downloading again.

- The cache lives at `~/.cache/cy-cloud` by default. Override it with the `CYCLOUD_CACHE_DIR` environment variable.
- Entries expire after **30 days**, and the cache is capped at **1 GB** by default. When the cap is exceeded, the oldest entries are removed first.
- Maintenance runs automatically. The CLI validates cache integrity at most once a day, and prunes expired or oversized entries after commands run.

Inspect or clear the cache at any time with the [`cache info`](#cy-cloud-cache-info) and [`cache clear`](#cy-cloud-cache-clear) commands. To turn off automatic maintenance, set `CYCLOUD_DISABLE_CACHE_MGMT=true`.

## <Icon name="sliders" /> Configuration

The CLI stores configuration and credentials under `~/.config/cy-cloud` (or `$XDG_CONFIG_HOME/cy-cloud` if that variable is set). Credentials are written with owner-only permissions.

The following environment variables adjust its behavior:

| Variable                     | Purpose                                                                                                                                                |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `CYPRESS_CLOUD_TOKEN`        | Authenticate with a personal access token. Takes precedence over stored credentials.                                                                   |
| `CYCLOUD_CACHE_DIR`          | Directory for the local replay cache. Defaults to `~/.cache/cy-cloud`.                                                                                 |
| `CYCLOUD_DISABLE_CACHE_MGMT` | Set to `true` to disable automatic cache validation and cleanup.                                                                                       |
| `CYCLOUD_URL`                | Point the CLI at a non-production Cypress Cloud API instance. Intended for Cypress internal development and testing; leave it unset to use production. |

## <Icon name="gauge" /> Rate limits

The Cloud CLI is available on all Cypress Cloud plans, and the same rate limit applies across every plan. Requests are limited **per user** to a default of **100 requests per hour**.

If you exceed the limit, requests return an error until the hour resets. Contact [support@cypress.io](mailto:support@cypress.io) if you need a higher limit.

Rate limits are separate from data availability. What data you can retrieve follows the same recording limits and [data retention](/cloud/account-management/billing-and-usage) policies as the rest of Cypress Cloud, so those limits do vary by plan. Runs beyond your retention window or usage limits are not accessible through the Cloud CLI.

## <Icon name="shield-halved" /> Restrictive VPNs and allowlists

The Cloud CLI communicates with the Cypress Cloud API at `https://cloud-api.cypress.io`. If you run the CLI from behind a restrictive VPN or firewall, add that host to your allowlist so the CLI can reach Cloud. Sign-in also uses `https://cloud.cypress.io`.

## <Icon name="robot" /> Use with AI agents

Because every command returns structured JSON and exposes its output schema with `--schema`, the Cloud CLI is a natural tool for AI coding agents. An agent can list failing tests, read errors, and pull replay timelines, then use that context to propose a fix in your editor.

The Cloud CLI and [Cloud MCP](/cloud/integrations/cloud-mcp) solve related problems in different ways:

- **Cloud MCP** is a remote server your AI client connects to. The agent selects Cypress tools automatically based on your prompt. It is the best fit for conversational, in-editor debugging.
- **The Cloud CLI** is a command-line tool for humans, scripts, and agents that run shell commands. It is the best fit for automation, custom tooling, and agents that work through a terminal.

Many teams use both. If your agent works through a terminal, give it the CLI. If it speaks MCP, connect it to Cloud MCP.

### Example prompts

If your agent runs shell commands (for example, Claude Code, Codex CLI, or Copilot CLI), point it at `cy-cloud` and describe the goal in plain language. A few prompts to start from:

> "Using the `cy-cloud` CLI, find the most recent failed run on `main` for project `abc123` and summarize why each failing test failed."

> "With `cy-cloud`, pull the replay timeline around the failure for test `<testId>`, including commands, network requests, and console logs, then look at my code and propose a fix."

> "Run `cy-cloud status` to confirm I am signed in, then list the failing specs in run 4021 of project `abc123` and save the first failure's screenshot to `./screenshots`."

> "Every morning, run `cy-cloud` to check the latest run on `main` for project `abc123`. If it failed, open a summary of the failing tests and their errors."
