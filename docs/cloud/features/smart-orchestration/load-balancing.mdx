# Load Balancing

:::info

## What is load balancing?

When you run tests
[in parallel](/cloud/features/smart-orchestration/parallelization) across
multiple CI machines, _how_ you split the work matters just as much as _how
many_ machines you use. If one machine is handed all the slow specs while the
others sit idle, you pay for every machine but only move as fast as the
slowest one.

**Load balancing** is how Cypress Cloud solves this. Cypress automatically
distributes your spec files across the available machines in your CI provider,
using data from previous runs to predict how long each spec will take and
assign work accordingly. The goal is simple: keep every machine busy until the
last spec finishes, so your run completes as fast as the hardware you're paying
for allows — with no manual configuration.

## Why load balancing matters

### For engineering leaders

- **Faster feedback, shipped sooner.** Balanced runs finish in the shortest
  time your machine count allows, so pull requests are verified sooner and
  developers spend less time waiting on CI.
- **Lower CI spend.** Idle machines are wasted money. By keeping every machine
  working until the run ends, load balancing extracts the full value of the CI
  resources you've already provisioned — often the difference between needing
  more machines and getting more out of the ones you have.
- **Predictable scaling.** As you add machines to keep up with a growing suite,
  Cypress redistributes work to match. You scale capacity without re-tuning how
  tests are split.
- **Zero maintenance overhead.** There are no hand-maintained spec lists,
  shard manifests, or "machine 3 is always slow" tickets. The work that would
  otherwise fall on your team is handled by Cypress Cloud automatically.

### For practitioners

- **No configuration required.** Load balancing is on by default whenever you
  run with `--record --parallel`. There's nothing to set up and nothing to
  keep up to date.
- **Self-correcting as your suite changes.** Add a spec, delete a spec, or
  watch one get slower over time — Cypress keeps balancing correctly because it
  learns from recent run history rather than a static configuration you have to
  maintain.
- **No flaky, hand-tuned sharding.** You don't have to guess which specs to
  group together or rebalance shards by hand. Cypress assigns specs one at a
  time to whichever machine is free next, so a single long spec can't strand a
  machine.
- **Browser-aware accuracy.** Predictions are made per browser, so runs stay
  balanced even when the same spec behaves differently across Chrome, Firefox,
  Edge, or Electron.

## How it works

### Balance strategy

Cypress calculates which spec file to run on each machine based on the data
collected from previous runs. Rather than splitting specs evenly by count,
Cypress estimates the duration of each spec and distributes them so that the
total run finishes as quickly as possible.

As more and more tests are recorded to the Cloud, Cypress can better predict
how long a given spec file will take to run. To prevent irrelevant data from
affecting the duration prediction, Cypress doesn't use old historical run data
regarding the spec file — keeping estimates accurate as your application and
tests evolve.

Because specs are handed out one at a time to whichever machine becomes
available next, work naturally flows to wherever there's free capacity. This
also means the run order of spec files is not guaranteed when parallelized.

### Spec duration history analysis

With a duration estimation for each spec file of a test run, Cypress can
distribute spec files to available CI resources in descending order of spec run
duration. In this manner, the most time-consuming specs start first, which
minimizes the overall test run duration and keeps faster specs available to
fill in any remaining gaps at the end of the run.

:::info

Duration estimation is done separately for every browser the spec file is tested
against. This is helpful since performance characteristics vary by browser, and
therefore it is perfectly acceptable to see different duration estimates for
each browser a spec file was tested against.

:::

## Get started

Load balancing works automatically once your project is
[recording to Cypress Cloud](/cloud/get-started/setup#Setup) and running in
parallel. Add the [`--parallel`](/app/references/command-line#cypress-run-parallel)
flag to your [`cypress run`](/app/references/command-line#cypress-run) command:

```shell
cypress run --record --key=abc123 --parallel
```

From there, Cypress Cloud takes care of distributing your specs across every
available machine — no further setup needed.
