# Project Analytics & Insights

Every time you run `cypress run --record`, Cypress Cloud captures the result.
Project Analytics turn that history into answers to the two questions every team
asks about its test suite:

- **Is my CI healthy and fast?** Are runs passing, how long do they take, and is
  the suite growing in a sustainable way?
- **What specifically is broken, slow, or flaky, and who or what is
  responsible?** Which tests, specs, branches, browsers, or commits are dragging
  the suite down?

Instead of digging through individual runs, you get trends and rankings across
your whole project so you can spot regressions early, prioritize the fixes that
matter most, and prove that investments like parallelization are paying off.

Project Analytics live under each project and are scoped to a **single
project**. (For a cross-project, organization-wide view, see
[Enterprise Reporting](/cloud/features/analytics/enterprise-reporting).)

## How analytics is organized

There are eight reports, grouped into two families based on the question they
answer.

| Family                          | Answers                                                       | Reports                                                             |
| ------------------------------- | ------------------------------------------------------------- | ------------------------------------------------------------------- |
| **Trend reports**               | _"Is my CI healthy and fast?"_                                | Run status, Run duration, Past builds run duration, Test suite size |
| **Ranking & breakdown reports** | _"What's broken/slow/flaky, and who or what is responsible?"_ | Top failures, Slowest tests, Most common errors, Flaky tests        |

- **Trend reports** are time-series views of how your run history behaves over
  time, with a broader default window (up to a year). Use them to monitor the
  overall direction of the suite.
- **Ranking & breakdown reports** are _"which tests are worst"_ views with a
  shorter default window (7 days), richer per-test filtering, a **View by**
  toggle, and drill-ins. This is where you slice by browser, OS, committer,
  spec, or Cypress version, then drill into individual tests with
  [Test Replay](/cloud/features/test-replay) and Jira issue linking.

Every report shares a common filter bar, exports its results to a
comma-separated values (CSV) file (via the download icon to the right of the
filters), and keeps filter state in the URL so views are easy to share.

## Filters & controls

Every report shares a common filter bar, but the set of available filters
depends on the report family.

### Available on every report

| Filter         | What it does                                                                                                   |
| -------------- | -------------------------------------------------------------------------------------------------------------- |
| **Branch**     | Multi-select, searchable. Limit results to specific git branches.                                              |
| **Tags**       | Multi-select of run tags (with colors, inline-editable). Includes a **match mode**: `ANY` (OR) vs `ALL` (AND). |
| **Time range** | Date-range picker with common presets, or choose a custom range. Results can go back up to 1 year.             |

### Additional controls on Trend reports

- **Time interval**: how the chart buckets results over time (for example,
  Daily, Weekly, or Monthly).
- **Build interval**: on the by-build view, how many runs each point on the
  chart represents. Grouping by 1 plots every run individually, while larger
  groupings aggregate that many consecutive runs to smooth the trend.

### Additional controls on Ranking & breakdown reports

- **View by**: regroup the ranking by Test Case, Spec File, Tag, or Branch.
- Slice results by **Run Group**, **Committer**, **Spec File**, **Browser**,
  **Cypress Version**, **Operating System**, and (where applicable) **Status**.

## Trend reports

### Run status

Track overall run volume and pass/fail health trends. Each run represents a
single invocation of `cypress run --record` for this project, whether in CI or
on a local machine.

- **What it shows:** A stacked bar chart of run outcomes over time (or by build
  number).
- **Measures:** Total runs, average runs per day, passed count, and failed
  count, plus a breakdown of every run status: Passed, Failed, Running, Timed
  out, Errored, No tests, Over limit, and Canceled. Includes the delta versus
  the previous period.
- **Use it to:** Track overall run volume and pass/fail health trends.

**Example workflow:** After a busy week of merges, you open Run status filtered
to your `main` branch and notice failed runs spiked over the weekend. You switch
the time interval to daily to pinpoint the day it started, then drill into that
day's runs to find the change that broke the build and revert it.

### Run duration

See how long your runs take and how much parallelization is helping. Only
passing runs are included here, since failing or errored runs can sway the
average away from its typical duration.

- **What it shows:** The run duration trend over time.
- **Measures:** Average run duration, typical concurrency (the median number of
  machines / parallel agents used), and **time saved from parallelization**.
- **Use it to:** See how long runs take and how much parallelization is helping.

**Example workflow:** Runs have started to feel slow. You open Run duration for
the last 30 days and see average duration creeping up while time saved from
parallelization stays flat, a sign the suite is growing faster than your
parallelization can absorb. You add CI machines (or raise concurrency) and
watch the trend flatten back out.

### Past builds run duration

The same duration metrics as the Run duration report, but plotted **per build
number** rather than per time bucket. This is useful when you want to inspect
specific builds rather than a smoothed trend.

- **What it shows:** Duration metrics plotted per build number instead of per
  time bucket.
- **Measures:** Average run duration, typical concurrency (mean machines), and
  time saved from parallelization.
- **Use it to:** Inspect duration build-by-build rather than smoothed over time.

**Example workflow:** A single nightly build took twice as long as usual, but a
smoothed daily average hides it. You open Past builds run duration, set the
build interval to 1 so every run is plotted individually, find the exact build
where duration jumped, and open that run to see which spec ran long.

### Test suite size

Watch how your test suite grows (or shrinks) across releases. It calculates the
average number of test cases executed per run for the given time period, and
excludes runs that errored or timed out since they don't accurately represent
the size of your suite.

- **What it shows:** Growth of the test suite over time.
- **Measures:** Total test cases, number of spec files, and the delta change.
- **Use it to:** Watch how the suite grows (or shrinks) across releases.

**Example workflow:** Before a release, you open Test suite size and notice the
total test count dropped last week even though no specs were intentionally
removed, a hint that tests were accidentally skipped or deleted. You track down
the affected spec files and restore the missing coverage before shipping.

## Ranking & breakdown reports

### Top failures

Find the worst offenders dragging down your suite.

- **What it shows:** A horizontal bar chart ranking the most-failing tests or
  specs, with drill-in.
- **Measures:** Median failure rate (%) and number of failures.
- **Drill-ins:** Latest failures, error details, Jira issue linking, and
  [Test Replay](/cloud/features/test-replay).
- **Use it to:** Find the worst offenders dragging down your suite.

**Example workflow:** CI has been red all week. You open Top failures for the
last 7 days, sort by failure rate, and find one test failing in most runs. You
drill into its latest failures and watch the Test Replay to see the root cause,
then file a Jira issue straight from the report so it gets assigned and fixed.

  src="/img/cloud/features/analytics/dashboard-analytics-top-failures-graph.png"

/>

### Slowest tests

Target the biggest contributors to your run time.

- **What it shows:** A horizontal bar chart ranking the slowest tests or specs,
  with a chart-range selector to filter by a duration band.
- **Measures:** Median duration and number of tests / specs.
- **Use it to:** Target the biggest contributors to run time.

**Example workflow:** Your run-duration trend is climbing and you want quick
wins. You open Slowest tests, group by spec file, and focus on the handful of
specs with the highest median duration, then split or optimize them to bring
overall run time back down.

<DocsImage
  src="/img/cloud/features/analytics/dashboard-analytics-slowest-tests-graph.png"
  alt="Cloud Analytics Slowest Tests Graph Screenshot"
/>

### Most common errors

Spot systemic error patterns rather than individual test failures.

- **What it shows:** A color breakdown / stacked bar of error classes by
  frequency; click an error class for detail.
- **Measures:** Number of failures and number of distinct error types, with
  proportions.
- **Drill-ins:** Error-class details and test history.
- **Use it to:** Spot systemic error patterns rather than chasing individual
  test failures.

**Example workflow:** Dozens of tests start failing at once. Instead of triaging
each one, you open Most common errors and see a single error class (such as a
network timeout) accounts for most of the failures, pointing to one systemic
cause to fix rather than many separate bugs.

<DocsImage
  src="/img/cloud/features/analytics/dashboard-analytics-common-errors.png"
  alt="Cloud Analytics Most Common Errors Screenshot"
/>

### Flaky tests

Track flakiness trends and triage your most unstable tests. For more on how
Cypress identifies and manages flake, see
[Flaky Test Management](/cloud/features/flaky-test-management).

- **What it shows:** A time-series of flakiness, color-coded by severity.
- **Measures:** Overall flakiness (%) and counts of High / Medium / Low severity
  flaky tests. Buckets auto-adjust (hourly or daily) to the chosen range.
- **Drill-ins:** Latest flakiness, error details, Jira issue linking, and
  [Test Replay](/cloud/features/test-replay).
- **Use it to:** Track flakiness trends and triage the most unstable tests.

**Example workflow:** At the end of a sprint you open Flaky tests and see overall
flakiness rising, with a few new high-severity flaky tests. You drill into each
with Test Replay to diagnose the nondeterminism, link Jira issues to assign the
fixes, then watch the high-severity count fall in the weeks that follow.
