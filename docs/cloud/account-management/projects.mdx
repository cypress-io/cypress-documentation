# Projects

:::info

##### <Icon name="question-circle" color="#4BBFD2" /> What you'll learn

- How to set up a project to record test runs
- How to manage project settings
- How to transfer ownership of a project
- Best practices for multi-repo implementation
- How to integrate with source control providers, issue management, and team communication tools

:::

A **project** is how Cypress Cloud organizes everything it knows about a single
test suite: its recorded runs, analytics, settings, and integrations. Connecting
a project unlocks the value of Cypress Cloud. Instead of test results that
vanish in CI logs, you get a durable, searchable history of every run, the
tools to make those runs faster and cheaper, and the integrations to surface
results where your team already works.

This page walks through each project-level feature and, just as importantly,
**why you'd use it**, so you can decide which settings deliver the most value
for your team.

## Identification

Cypress uses your [projectId](#Project-ID) and [Record Key](#Record-key)
together to uniquely identify projects. This two-part identity is what keeps
your data trustworthy: the `projectId` tells Cypress Cloud _which_ project a run
belongs to, while the record key proves the run is _authorized_ to write to it.
Together they ensure your run history stays accurate and can't be polluted by
unauthorized recordings. This also protects your bill: because Cypress charges
based on the number of recorded tests, keeping your record key private prevents
unauthorized runs from consuming your test allotment and driving up usage.

### Project ID

Once you set up your project to record, we generate a unique `projectId` for
your project and automatically insert it into your Cypress configuration file.

#### The `projectId` is a 6 character string in your Cypress configuration

:::cypress-config-example

```ts
{
  projectId: 'a7bq2k',
}
```

:::

This helps us uniquely identify your project. If you manually alter this,
**Cypress will no longer be able to identify your project or find the recorded
builds for it**.

If you're using source control, we recommend that you check your Cypress
configuration file, including the `projectId`, into source control. If you don't
want your `projectId` visible in your source code you can set it as an
environment variable using the name `CYPRESS_PROJECT_ID`. The exact mechanism
for doing so depends on your system but could be something like:

```shell
export CYPRESS_PROJECT_ID={projectId}
```

### Record key

The record key is used to authenticate that your project is _allowed_ to record
tests to Cypress Cloud. As long as your record key stays _private_, no one will
be able to record test runs for your project - even if they have your
`projectId`.

Think of your record key as the key that enables you to _write and create_ runs.
However, it has nothing to do with being able to _read or see_ runs once they
are recorded.

The ability to create and revoke keys is what gives you control over who can
contribute to your run history. Issuing separate keys per CI environment, then
rotating or deleting any key that leaks, lets you cut off unauthorized recording
instantly without disrupting your other pipelines.

:::caution

<strong>Keep your record key private</strong>

Anyone with both your `projectId` and record key can record runs to your project
in Cypress Cloud. Keep the record key out of reach of anyone outside your team,
for two reasons:

1. **It can inflate your bill.** Cypress bills based on the number of recorded
   tests, so unauthorized runs can consume your test allotment and push you over
   your plan's limit.
2. **It lets others write data to your project.** Someone could record runs that
   capture content you never intended, such as videos of visits to unrelated
   websites, polluting your run history.

If your record key is ever exposed, [delete it](#Delete-record-key) and
[create a new one](#Create-new-record-key). A deleted key is permanently invalid:
any run that uses it will fail to record.

To avoid exposing the key in the first place, store it as an environment variable
rather than in your source code. See
[environment variables](/app/continuous-integration/overview#Environment-variables)
for details.

:::

Once you're set up to record test runs, we automatically generate a _Record Key_
for the project.

#### A record key is a GUID that looks like this

```text
f4466038-70c2-4688-9ed9-106bf013cd73
```

You can create multiple Record Keys for a project, or delete existing ones from
[Cypress Cloud](https://on.cypress.io/cloud).

<DocsImage
  src="/img/cloud/projects/record-keys-in-project-settings-dashboard.jpg"
  alt="Record key in project settings"
/>

You can also find your Record Key inside of the _Settings_ tab in the Cypress
App.

<DocsImage
  src="/img/cloud/projects/record-key-desktop-gui.jpg"
  alt="Record Key in Configuration Tab"
/>

#### Create new record key

1. Go to your organization's projects page.
2. Select the project you want to add a record key to.
3. Go to the project's **Settings** page.
4. Find the **Record Keys** section.
5. Click **Create New Key**. A new key will be automatically generated for your
   project.
6. Copy the key and use it to record runs by passing it to
   [`cypress run`](/app/references/command-line#cypress-run) with the
   `--record --key <record-key>` flags, or by setting it as the
   `CYPRESS_RECORD_KEY` [environment variable](/app/continuous-integration/overview#Environment-variables)
   in your CI provider.

#### Delete record key

1. Go to your organization's projects page.
2. Select the project whose record key you want to delete.
3. Go to the project's **Settings** page.
4. Find the **Record Keys** section.
5. Click **Delete** beside the record key you want to delete.

## Access and visibility

These settings control who can see your project's runs and how you advertise
their status. Set them deliberately to match your audience, whether that's a
private team or the public.

### Access to Runs

Visit your project settings to see who has access to your project's runs.

Controlling access lets you match visibility to your goals. Make a project
**public** to showcase the health of an open-source library and let contributors
debug their own failures; keep it **private** to protect a commercial codebase so
only invited teammates can view runs, screenshots, and test code.

#### Public vs Private

- **Public** means that anyone can see the recorded test runs for the project,
  including people outside of your organization. It is similar to how public
  projects on GitHub are handled. Anyone who knows your
  `projectId`, whether or not they belong to your organization, will be able to
  see the recorded runs for public projects.

- **Private** means that only [users](/cloud/account-management/users)
  you invite to your
  [organization](/cloud/account-management/organizations) can see its
  recorded runs. Even if someone knows your `projectId`, they will not have
  access to your runs unless you have invited them. For finer-grained control,
  you can assign a private project to one or more
  [teams](/cloud/account-management/teams) so that only members of those teams,
  rather than everyone in the organization, can access its runs.

#### Change project access

1. Go to your organization's projects page.
2. Select the project you want to change access to.
3. Go to the project's **Settings** page.
4. Here you will see a section displaying **Access**. Choose the appropriate
   access you'd like to assign.
   <DocsImage
     src="/img/cloud/projects/access-to-runs.jpg"
     alt="access-to-runs"
   />

### README Badges

README badges allow you to increase visibility of your project's test status and
test count to other developers viewing your project's README file.

**Why it's valuable:** A badge turns your latest run into an at-a-glance signal
of quality right where people decide whether to trust or contribute to your
project. For open-source maintainers it builds confidence with potential users
and contributors; for any team it advertises test coverage and current health
without anyone having to open Cypress Cloud.

#### Create a README badge

:::caution

**Note**: README badges are currently only available for public projects.

:::

1. In your Cypress Cloud account, select the project for which you'd like to add
   a badge.
2. On the Project Settings page, scroll down to the README Badges section and
   click “Configure Badge”.
3. A configuration modal will appear. The Project ID will be pre-filled with the
   ID associated with the project you selected. You can choose to designate a
   specific branch, or leave this field blank to always use the latest build in
   the project.
4. Next, style your badge. Flat is the default styling and is most commonly
   used, but 5 style options are available.
5. Select the badge type to change the amount and type of information that's
   displayed. Simple status will show only whether tests are passing or failing.
   Detailed status will show the number of tests that were passed, failed, or
   skipped. Test count will show how many tests are included in your project.
6. Once you've selected all your settings, check out the preview and make sure
   everything looks just the way you like it.
7. Your badge is ready to be embedded. Copy the markdown at the bottom of the
   Configure Badge modal, and embed it in your project's README file for
   everyone to see!

## Optimizing your runs

These settings tune how Cypress Cloud runs and finalizes your suite so you get
feedback faster, spend less on CI, and never wait on a run that has stalled. They
matter most for large suites and parallelized CI pipelines.

### Smart Orchestration

Smart Orchestration is about getting answers faster and paying for less CI.
Rather than running every spec in a fixed order until the suite finishes, Cypress
Cloud uses the history of your runs to surface failures sooner and stop work that
no longer needs to happen.

Toggle Spec Prioritization to reduce feedback time by running previously failed test specs first. Enabling Auto Cancellation to reduce run duration by canceling the run when tests fail. Adjust the number of test failures before auto
canceling.

**Why it's valuable:** Spec Prioritization puts the specs most likely to fail at
the front of the line, so a developer learns their change is broken in the first
minute of a run instead of the last. Auto Cancellation stops a run once it has
seen enough failures to know it won't pass, since there's no reason to burn ten more
minutes of machine time proving a build is already red. Together they shorten the
feedback loop, free up CI machines for the next job, and cut the number of billed
tests on doomed runs.

These settings can significantly improve developer resolution time, reduce your monthly CI bill and free up CI resources. See our
[Spec Prioritization](/cloud/features/smart-orchestration/spec-prioritization)
and [Auto Cancellation](/cloud/features/smart-orchestration/run-cancellation) guides to learn more.

<DocsImage
  src="/img/cloud/features/smart-orchestration/spec-prioritization-active.png"
  alt="Run completion delay settings"
/>

### Parallelization

Parallelization splits your specs across multiple machines so a suite that takes
30 minutes on one machine can finish in a few. The settings here exist to keep
that distributed run reliable, making sure every machine's results are counted
in a single, coherent run no matter when each one starts.

#### Run completion delay

Control how long the Cypress Cloud should wait for new groups to join a run before transitioning a run to a _completed_ status with the **Run completion delay** . This gives your test groups a grace period to join an existing run before the run is considered complete.

**Why it's valuable:** In a parallel setup, your CI machines rarely boot at the
exact same moment. Without a grace period, a fast-starting machine could finish
and close the run before a slower machine even joins, splitting one logical run
into two and skewing your results. The completion delay holds the run open just
long enough for every group to report in, so you get one accurate run instead of
fragmented, misleading data.

Use run completion delay when you have a distributed test setup where test groups might
not all start simultaneously. For example:

- CI jobs that might start at slightly different times
- [Parallelization](/cloud/features/smart-orchestration/parallelization) across multiple machines
- Flaky network conditions that might delay some test groups from connecting

The default is **60 seconds**. Increasing this value gives your test groups more time to join the run; decreasing it will make runs complete faster when no new groups are joining.

See our
[Parallelization guide](/cloud/features/smart-orchestration/parallelization#Run-completion-delay)
to learn more.

<DocsImage
  src="/img/cloud/projects/run-completion-delay.jpg"
  alt="Run completion delay settings"
/>

### Run Timeout

**Run timeout** is the number of minutes that Cypress Cloud will
wait to receive a signal from a run before marking the spec and run as **Timed Out**.
This is the maximum time Cypress Cloud will wait for activity from your tests before
considering them hung or failed.

**Why it's valuable:** A crashed CI agent or a hung test can otherwise leave a
run stuck "in progress" indefinitely, blocking merge checks and leaving your team
guessing whether tests are still running or quietly dead. The run timeout
guarantees a run always reaches a final state, so status checks resolve and your
pipeline keeps moving. Tightening it gives faster feedback when something hangs;
loosening it accommodates legitimately long tests without false alarms.

Use run timeout when you have long-running tests that might exceed the default. For
example:

- Very long-running end-to-end tests
- Tests that include operations that take significant time (such as large file uploads)
- Operations with unpredictable completion times

The default is **90 minutes**. You can shorten this for faster feedback when a run has stopped sending data.

<DocsImage
  src="/img/cloud/projects/run-timeout-setting.jpg"
  alt="Run timeout setting"
/>

#### Run completion delay vs. Run timeout

These settings work together to balance giving your tests enough time to complete
while not waiting indefinitely for runs that have failed or hung:

- **Run completion delay** (seconds): How long to wait for new test groups to join
  after all currently known groups complete. Applies only during that waiting period.
- **Run timeout** (minutes): How long to wait for any activity from a run before
  marking it timed out. Applies throughout the entire run.

### Test Replay

[Test Replay](/cloud/features/test-replay) captures a full snapshot of your
application (DOM, styles, network traffic, console logs, and more) so you can
debug CI failures as if you were there. Because that capture happens during
recording, the **Test Replay** section of your project settings gives you two
controls over it:

- **Test Replay recording.** Test Replay is on by default. Toggle it off to stop
  capturing replay data for the project entirely. Keep in mind that captured
  replays, including any test data they contain, are visible to everyone with
  access to the project. See
  [Opt-out of Test Replay](/cloud/features/test-replay#Opt-out-of-Test-Replay).
- **Canvas elements.** Capturing `<canvas>` elements can be resource-intensive. If
  your application renders many or large canvases and you notice slower recording,
  disable this toggle, which sits just below the main Test Replay toggle, to skip
  canvas capture while keeping the rest of Test Replay. See
  [Canvas Elements](/cloud/features/test-replay#Canvas-Elements).

<DocsImage
  src="/img/cloud/features/test-replay/cypress-test-replay-canvas-elements-toggle-cloud.png"
  alt="Test Replay canvas toggle in project settings"
/>

## Managing your project

These actions change who owns a project or remove it entirely. Because they
affect a project's whole run history, they're handled separately from the
day-to-day settings above.

### Transfer ownership

Transferring a project moves it, along with its entire run history, settings,
and integrations, to another user or organization without losing any data.
This is valuable when teams reorganize, when a project graduates from an
individual's account to a shared org, or when ownership changes hands: the
historical context that makes Cloud useful stays intact instead of starting
over from zero.

#### Transfer project to other user or organization

You can transfer projects that you own to another
[organization](/cloud/account-management/organizations) you are a part of
or to another user in the organization. Projects can only be transferred from
[Cypress Cloud](https://on.cypress.io/cloud).

1. Select your organization in the organization switcher.
2. Select the project you wish to transfer.
3. Go to the project's **Settings** page.
4. Scroll down to the **Transfer Ownership** section and click **Transfer
   Ownership**.
   <DocsImage
     src="/img/cloud/projects/transfer-ownership-button.jpg"
     alt="Transfer ownership button"
   />
5. Select the user or organization, then click **Transfer**.
   <DocsImage
     src="/img/cloud/projects/transfer-ownership-of-project-dialog.jpg"
     alt="Transfer Project dialog"
     width="80%"
   />

:::caution

<Icon name="exclamation-triangle" /> You are not allowed to transfer a project
to an organization that is currently enrolled in a [free
trial](/cloud/get-started/free-trial).

:::

#### Cancel project transfer

Upon transferring, you can cancel the transfer at any time by visiting the
organization's projects and clicking **Cancel Transfer**.

<DocsImage
  src="/img/cloud/projects/cancel-transfer-of-project.jpg"
  alt="Cancel pending transfer of project"
/>

#### Accept or reject transferred project

When a project is transferred to you, you will receive an email notifying you.
You will be able to accept or reject the transferred project by clicking the
notification in the sidebar and clicking 'Accept' or 'Reject'.

<DocsImage
  src="/img/cloud/projects/accept-or-reject-transfer-of-project.jpg"
  alt="Accept or reject a transferred project"
/>

### Delete Project

You can delete projects you own. This will also delete all of their recorded
test runs. Deleting projects can only be done from
[Cypress Cloud](https://on.cypress.io/cloud).

1. Select your organization in the organization switcher.
2. Select the project you want to remove.
3. Go to the project's **Settings** page.
4. At the very bottom of the Settings page click the **Remove Project** button.
   <DocsImage
     src="/img/cloud/projects/remove-project-dialog.png"
     alt="Delete project dialog"
     width="80%"
   />
5. Confirm that you want to delete the project by clicking **Yes, Remove
   Project**.

## Best Practices

A few habits keep your Cypress Cloud projects organized, secure, and
cost-effective as your team and test suite grow.

### Keep one project per test suite

Record all of a suite's runs to a single project tied to a single `.git`
repository. Consistent commit and branch data is what makes your run history,
analytics, and [flaky test detection](/cloud/features/flaky-test-management)
meaningful, so avoid pointing several unrelated suites at one project or
splitting one suite across many.

### Multi-repo Implementation

When Cypress tests reside in a repository separate from the application code, it creates a suboptimal Cloud experience - especially when considering the commit and branch interpretation for each git directory.

:::info

For any given Cypress project, all commits or branches associated to recorded runs should belong or relate to a single `.git` repository.

:::

If this is not possible, there are some measures to help organize your data:

1. Create two separate projects in Cypress Cloud, for example, **App Project** and **Test Project**

2. For the **App Project**:
   - Link the GitHub integration to the Application code repo
   - Override the `COMMIT_INFO_*` environment variables used by the test recording to reflect the state of the checked out Application code repo
   - Record any runs related to the Application code repo to **App Project**
   - Capture any Cypress config/tests repo related identifiers (short sha/release number) as tags decorated on the run

3. For the **Test Project**:
   - Link the GitHub integration to the Cypress config/tests repo
   - Ensure any runs recorded to **Test Project** are related to test code changes in the Cypress config/tests repo
     - The Application code repo state that the tests are executed against could likewise be captured as tags here, if necessary.

### Protect your record key

Store your [record key](#Record-key) as a CI secret or
[environment variable](/app/continuous-integration/overview#Environment-variables)
instead of committing it, and [rotate it](#Delete-record-key) if it is ever
exposed. This keeps unauthorized runs from polluting your history or consuming
the test allotment you are billed for.

### Commit your projectId to source control

Check your [`projectId`](#Project-ID) into your Cypress configuration so every
contributor and CI machine records to the same project. If you would rather not
expose it in source, set it as the `CYPRESS_PROJECT_ID` environment variable
instead.

### Match project access to your audience

Set your project [public or private](#Public-vs-Private) deliberately: public for
open-source suites you want contributors and users to see, private for commercial
code that should stay limited to invited teammates.

### Let Smart Orchestration manage CI time and cost

On the Business and Enterprise plans, enable
[Spec Prioritization and Auto Cancellation](#Smart-Orchestration) so failing runs
surface fast and stop early, cutting wasted CI minutes and billed tests on runs
you already know will fail.

## Third party integrations

You can integrate your project with many third party tools and edit those
settings from within the project settings page.

Integrations bring your test results to where your team already works, so no one
has to go looking for them. Connecting a **source control provider** posts run
status directly on your pull requests, so a failing test blocks a merge before
broken code reaches your main branch. An **issue management** integration turns a
failure into a tracked ticket, complete with a link back to the failing run, in
a couple of clicks. **Team communication** integrations push run results into the
channels you already watch, so the right people hear about a failure in seconds
instead of discovering it hours later. In short, integrations close the loop
between a test failing and your team acting on it.
