# Data Storage and Controls

Cypress Cloud captures the context around your test runs so you can debug
failures faster, collaborate with your team, and gain insight into the health
of your test suite over time. We also know that test runs can surface sensitive
information, and that you need to stay in control of what leaves your
environment.

This page explains exactly what data Cypress Cloud stores and how it's used, then
walks through the controls you can use to limit, mask, or prevent that data from
being captured, so you can get the full value of Cypress Cloud while meeting your
team's privacy, security, and compliance requirements.

:::info

##### <Icon name="question-circle" color="#4BBFD2" /> What you'll learn

- What data is stored in Cypress Cloud and how it is used
- How to control what data is sent to Cypress Cloud
- How to mask data in screenshots and videos
- How to control what is shown in the Cypress Command Log and network requests

:::

Please refer to the [Cypress Cloud Terms of Use](https://cypress.io/terms-of-use) and our [Security & Compliance](https://on.cypress.io/security/) guide for more specifics around reasonable use of Cypress Cloud, data storage and security at Cypress.

:::

## Types of Data

Understanding what Cypress Cloud stores is the first step in deciding which
controls are right for your team. Each type of data exists to make debugging and
collaboration easier, and each can be limited or turned off using the
[Data Controls](#Data-Controls) below.

When you run Cypress via `cypress run` and pass the `--record` flag, the following data is stored in Cypress Cloud for every run. This is what powers your run history, surfaces flaky tests, and lets your whole team see results in one place:

- Standard output in the terminal
- Test results
- Test definitions
- Cypress configuration (minus Cypress environment variables)
- Screenshots
- Videos
- OS environment variables related to CI and git information

When [Test Replay](/cloud/features/test-replay) capturing is enabled, the additional data outlined below is sent to Cypress Cloud. This is what makes it possible to replay and inspect a failed run exactly as it happened in CI, with no more re-running tests locally to reproduce a problem. [See our guide on disabling Test Replay.](/cloud/features/test-replay#Opt-out-of-Test-Replay)

- The rendered DOM and CSS styles for the application under test
- Cypress commands and events represented in the Command Log
- Network traffic within your application under test
- Browser console logs

Cypress Cloud does not capture anything related to the code of your application under test or the code within any repositories that are associated with your project.

When [Cloud AI](/cloud/faq#What-is-Cloud-AI) is enabled, test content may be shared with a session-bound AI model to enhance your test management experience. Your data is not used to train AI models and will not be shared between customers.

You, the customer, own and are responsible for your test content. We trust that customers will make decisions about the appropriateness of data to use in testing and will avoid use of PII, PHI, or other types of protected information. Cypress makes commercially reasonable efforts to protect confidentiality of information that is provided to Cypress Cloud.

## Data Controls

You decide what data Cypress Cloud sees. The controls below let you tune the
balance between richer debugging context and tighter data privacy, from
restricting who can view a project, to masking individual values, to opting out
of data capture entirely. Most teams use a combination of these to keep sensitive
data out of the Cloud while still benefiting from features like Test Replay.

If you'd like more granular control over what data is sent to Cypress Cloud, please [contact us](https://www.cypress.io/contact/) to discuss your needs.

### Project access controls

**Why it matters:** Controlling who can see a project is your first line of
defense for keeping run data confidential within your organization.

Cypress Cloud offers controls around the visibility of a project. You can set a project to be public or private. [See project access control settings.](/cloud/account-management/projects#Public-vs-Private)

Setting a project to private means that only the users explicitly invited to the organization will have visibility into the project, its runs, and any data captured by Cypress.

For even finer-grained control, [teams](/cloud/account-management/teams) let you decide which users can see which projects _within_ your organization. Rather than every member seeing every private project, you assign projects to a team and add only the relevant users to it, so people see just the projects they need. This is useful when:

- Multiple business units or clients share one Cypress Cloud organization and their projects (and the data captured in them) should stay isolated from one another.
- Certain projects contain especially sensitive applications or data that only a specific group should be able to view.

By default, all users belong to a built-in _Everyone_ team that has access to all projects. To enforce a stricter, need-to-know model, create teams aligned to your structure, assign each team only the projects it should access, and remove the _All projects_ assignment from the _Everyone_ team. See [Manage teams](/cloud/account-management/teams) for setup steps and best practices. Note that [public projects](/cloud/account-management/projects#Public-vs-Private) are outside the scope of teams and remain visible to everyone in the organization.

### Test Replay controls

**Why it matters:** Test Replay captures the most detail about your application,
including the DOM, network traffic, and console logs. Disabling it gives you a single switch to
stop all of that data from leaving your environment when the trade-off isn't right
for a given project.

#### Disable Test Replay

You can disable capturing data in Test Replay per project by following [this guide](/cloud/features/test-replay#Opt-out-of-Test-Replay). This means Test Replay will not be available for viewing or debugging in the project, but it guarantees Cypress Cloud never captures the following data:

- The rendered DOM and CSS styles for the application under test
- Cypress commands and events represented in the Command Log
- Network traffic within your application under test
- Browser console logs

If you want to keep Test Replay enabled but exclude what your application renders to `<canvas>` elements, you can disable canvas capture on its own. Open project settings in Cypress Cloud and toggle off the canvas option in the Test Replay section, which sits below the main Test Replay recording toggle. This keeps the rest of Test Replay available while preventing canvas content from being captured. See [Canvas Elements](/cloud/features/test-replay#Canvas-Elements) for details.

### Screenshot controls

**Why it matters:** Screenshots are invaluable for spotting visual regressions
and understanding failures at a glance, but a single screenshot can also expose
information you don't want captured. These controls let
you keep the debugging value while hiding what shouldn't be captured.

#### Blackout elements in screenshots

Cypress offers an option to black out (mask) an array of elements from screenshots. [See screenshot API docs.](/api/commands/screenshot#Arguments) This allows you to black out specific elements on screenshots so that they will not be visible in the locally saved screenshots or to any users that have access to the run in Cypress Cloud.

Add a control to your [support file](/app/references/configuration#Testing-Type-Specific-Options) to set the default blackout options for all screenshots.

```js
before(() => {
  Cypress.Screenshot.defaults({
    // blackout all elements with 'mask-cy' attribute
    blackout: ['mask-cy'],
  })
})
```

Add a control for a specific screenshot to blackout specific elements.

```js
it('admin page loads', () => {
  cy.visit('/admin')
  cy.screenshot({
    blackout: ['.user-name'],
  })
})
```

#### Remove Cypress Command Log from screenshots

The Cypress Command Log can include typed values and assertion details. If you'd
rather capture only the application itself, Cypress lets you hide the Command Log
from screenshots so that no information from that area is captured. See the [`cy.screenshot()`](/api/commands/screenshot#Arguments) command for more information.

Add a control to your [support file](/app/references/configuration#Testing-Type-Specific-Options) to set the default capture options for all screenshots.

```js
before(() => {
  Cypress.Screenshot.defaults({
    capture: 'viewport', // or fullPage
  })
})
```

Add a control for a specific screenshot to only capture the application under test.

```js
it('sign in page', () => {
  cy.visit('/sign-in')
  cy.screenshot({
    capture: 'viewport', // or fullPage
  })
})
```

### Video controls

**Why it matters:** Videos make it easy to see what happened during a run, but you
may not want every run's video stored long-term. This control lets you keep videos
only when they're useful, for example for failing runs, and discard the rest.

#### Disable video upload

You can delete videos that are recorded when you are passing `video: true` by following [this guide to control what is uploaded to Cypress Cloud](/app/guides/screenshots-and-videos#Control-which-videos-to-keep-and-upload-to-Cypress-Cloud).

### Cypress Command Log controls

**Why it matters:** The Command Log is one of Cypress's most powerful debugging
features, but it can render sensitive values like passwords or tokens. These
controls let you keep using it while preventing specific data from appearing in
screenshots and videos.

#### Turn off Cypress Command Log rendering

You can disable showing the entirety of the Cypress Command Log as well as the Application under Test (AUT) controls in your tests to prevent all test content from being shown and recorded in any screenshot or video by passing `--no-runner-ui` via the CLI while running Cypress tests. See [Cypress Command Line options](/app/references/command-line#Options). This means that the AUT will be the only thing that is rendered while running tests, so the only thing that is captured in screenshots and videos.

Note that when `--no-runner-ui` is set, data from the Cypress Command Log will still be sent to Cypress Cloud to be generated and displayed in Test Replay if it is enabled. See [Disable Test Replay](/cloud/features/test-replay#Opt-out-of-Test-Replay) for more information.

#### Mask data in the Cypress Command Log

When you want to keep the Command Log but hide specific values, Cypress gives you
several targeted ways to mask them. Note that when these strategies are used, data within test definitions will still be sent to Cypress Cloud.

##### `log: false` for Cypress commands

You can pass `log: false` as an option to Cypress-specific commands in your tests to prevent the data from being shown visibly in the Cypress Command Log and being shown in any captured screenshots or videos.

##### Disable logging of XHR/Fetch requests

By default, Cypress logs all requests that match any [`cy.intercept()`](/api/commands/intercept) in the Cypress Command Log, as well as all XMLHttpRequests and fetch requests. When those requests carry sensitive payloads, you can use `cy.intercept()` to disable these logs by passing `{ log: false }` in the second parameter. See the [`cy.intercept()`](/api/commands/intercept#Disabling-logs-for-a-request) command for more information.

```js
beforeEach(() => {
  // disable Cypress's default behavior of logging all XMLHttpRequests and fetches
  cy.intercept({ resourceType: /xhr|fetch/ }, { log: false })
})
```

### Network requests controls

**Why it matters:** Some hosts, such as analytics, third-party APIs, and internal services,
may handle data you don't want captured or contacted during a run. Blocking them
keeps that traffic out of Cypress Cloud and keeps your tests focused on the
application under test.

Cypress allows users to block requests from specified hosts. [See our blockHosts configuration guide.](/app/references/configuration#blockHosts)

### Cloud AI controls

**Why it matters:** Cloud AI enhances your test management experience, and you stay
in control of whether it's used. Admins can turn it on or off for the whole
organization with a single setting.

[Cloud AI](/cloud/faq#What-is-Cloud-AI) is enabled by default for all users on any Cloud plan.

Organization admins and owners can enable and disable the AI capabilities for their entire organization from the organization settings.
